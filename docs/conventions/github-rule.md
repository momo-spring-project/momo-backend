## 1. 커밋 규칙

| **타입**       | **설명**             |
|--------------|--------------------|
| **feat**     | 새로운 기능 추가          |
| **hotfix**   | 치명적 버그 수정          |
| **fix**      | 단순 코드 수정           |
| **refactor** | 코드 리팩토링 (기능 변화 없음) |
| **move**     | 파일/폴더 이동 및 구조 정리   |
| **del**      | 기능/파일 삭제           |
| **test**     | 테스트 코드 작성/수정       |
| **gitfix**   | `.gitignore` 수정    |
| **script**   | 빌드/패키지/환경 설정 관련 수정 |
| **rename**   | 파일 및 코드 이름 수정      |

> 💡 커밋 메시지는 반드시 작업 타입 + 간단한 설명 형식으로 작성합니다.
>
>
> 예: `feat: User 엔티티 및 레포지토리 생성`
>

---

## 2. 브랜치 규칙

- **개인 브랜치명**: `feat/기능-버전`
    - 예시: `feat/login-v2`, `feat/user-v1`, `feat/notification-v3`
- 브랜치는 반드시 **기능 단위 및 버전단위**로 생성하고, 완료되면 PR로 `dev` 브랜치에 병합합니다.

---

## 3. Pull Request 규칙

- **PR 작성 시 템플릿 사용**

```markdown
## #️⃣ 작업 한줄 요약

ex) feat: 유저 도메인 엔티티 및 레포지토리 생성

## 📝 작업 내용

- 이번 PR에서 작업한 내용을 간략히 설명해주세요
- 필요한 경우 이미지 첨부 가능

## 💬 리뷰 요구사항 (선택)

- 리뷰어가 특별히 확인해야 할 부분이 있다면 작성해주세요
- ex) 메서드 XXX의 네이밍이 적절한지 확인 부탁드립니다
```

- **PR 검토 규칙**
    - 최소 **2명 이상의 승인 필요**
    - PR 작성자는 2명 이상의 승인을 받은 후 머지 가능
    - 승인 후 `dev` 브랜치로 머지 진행

---

## 4. 기타 규칙

- `main` 브랜치는 항상 **배포 가능한 상태**를 유지합니다.
- 불필요한 파일/코드는 머지 전 반드시 정리합니다.
- 규칙에 없는 상황은 **팀원들과 합의 후 적용**합니다.
